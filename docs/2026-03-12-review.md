### ⏸️ C-09. 바우처-주문 상태 업데이트 비원자적 처리 — 보류
> **쉬운 설명**: 바우처 비밀번호를 설정하면 "바우처 상태"와 "주문 상태" 두 가지를 동시에 바꿔야 합니다. 그런데 지금은 하나씩 따로 바꾸고 있어서, 첫 번째는 성공하고 두 번째가 실패하면 **데이터가 엉킵니다**. 은행 송금에 비유하면, A 계좌에서 돈은 빠졌는데 B 계좌에는 안 들어간 상태가 될 수 있는 것입니다. 두 변경을 하나로 묶어서 "둘 다 성공 또는 둘 다 실패"로 만들어야 합니다.

- **발견**: 팀원 1 (논리), 팀원 3 (엣지케이스)

- **상태**: ⏸️ **보류** — 3연속 실패가 동시에 일어나야 하는 극히 드문 케이스. 서비스 규모 확대 시 처리 예정

## 💡 개선 제안 (Suggestion)

### S-01. 바우처 비밀번호 엔트로피 강화
> **쉬운 설명**: 바우처 비밀번호가 4자리 숫자(0000~9999)라서 경우의 수가 1만 개. 임시 비밀번호는 3자리로 1천 개. 자물쇠 번호를 더 길게 만들면 보안이 크게 올라갑니다.

- 사용자 비밀번호 4자리(10,000 조합) -> 6자리 권장
- 임시 비밀번호 3자리(1,000 조합) -> 6자리 영숫자 권장

### S-02. `as unknown as` 캐스팅 15곳 이상 개선
> **쉬운 설명**: 데이터베이스에서 가져온 데이터의 형태를 코드에서 "이렇게 생겼을 거야"라고 억지로 지정하는 곳이 15군데 이상 있습니다. 나중에 데이터 구조가 바뀌면 에러가 발생할 수 있어서, 자동으로 형태를 확인하는 방식으로 바꾸는 게 좋습니다.

- Supabase JOIN 결과에 타입 안전한 헬퍼 함수 도입 또는 generated types 활용

### S-03. API Route 보일러플레이트 28개+ 파일 중복
> **쉬운 설명**: 서버 기능을 처리하는 28개 이상의 파일에서 거의 동일한 코드가 반복됩니다. 하나를 고치면 28군데를 다 고쳐야 해서 유지보수가 힘듭니다. 공통 코드를 한 곳에서 관리하면 좋습니다.

- JSON 파싱 try-catch, 응답 구조 등을 공통 미들웨어/래퍼로 추출

### S-04. eslint-disable 3건 해결
> **쉬운 설명**: 코드 품질 검사 도구(ESLint)가 "이 부분에 문제가 있을 수 있다"고 경고하는데, 3곳에서 그 검사를 강제로 꺼놨습니다. 경고를 무시하는 대신 근본적으로 해결하는 게 좋습니다.

- `AdminFaqClient`, `MemberDetailModal`, `AdminNoticesClient`에서 `react-hooks/exhaustive-deps` 비활성화

### S-05. FeeMode vs FeeType 타입 중복
> **쉬운 설명**: 같은 개념("수수료 방식")을 표현하는 이름이 파일마다 다릅니다. 한쪽은 FeeMode, 다른 쪽은 FeeType. 같은 것을 다른 이름으로 부르면 헷갈리므로 하나로 통일해야 합니다.

- `ProductDetailClient`의 로컬 `FeeMode` 타입과 `types/index.ts`의 `FeeType`이 동일

### S-06. Pretendard 폰트 CDN -> next/font/local 전환
> **쉬운 설명**: 글꼴(폰트)을 외부 서버에서 매번 불러오고 있는데, 이러면 페이지 로딩 시 글꼴이 깜빡이면서 바뀌는 현상(FOUT)이 생길 수 있습니다. 글꼴을 우리 서버에 내장하면 더 빠르고 깔끔합니다.

- 성능 최적화 및 FOUT 방지

### S-07. `--font-mono` 변수가 Pretendard로 설정
> **쉬운 설명**: "고정폭 글꼴"(코드처럼 글자 너비가 동일한 글꼴)이 일반 글꼴(Pretendard)로 잘못 설정되어 있습니다. 디자인 문서와 실제 설정이 다릅니다.

- DESIGN-SYSTEM.md와 불일치, monospace 폰트로 설정 필요

### S-08. 관리자 레이아웃 전체 "use client"
> **쉬운 설명**: 관리자 페이지 전체가 "브라우저에서만 실행" 모드로 되어있어서, 서버에서 미리 화면을 그려두는 **속도 최적화 혜택을 못 받고 있습니다**. 필요한 부분만 브라우저 모드로 바꾸면 페이지 로딩이 빨라집니다.

- `(admin)/layout.tsx`가 전체 클라이언트 컴포넌트로 SSR 이점 상실

### S-09. getCategories() 캐싱 없이 3개 레이아웃에서 매번 호출
> **쉬운 설명**: 카테고리 목록(사이드바 메뉴)을 3개 페이지에서 매번 데이터베이스에 새로 요청합니다. 카테고리는 자주 안 바뀌니까 한번 가져오면 잠시 저장해두고 재사용하는 게 효율적입니다.

- React cache 또는 unstable_cache 활용 권장

### S-10. X-Forwarded-For 헤더 신뢰 문서화
> **쉬운 설명**: 관리자 접속 제한을 IP 주소로 하고 있는데, Vercel이 아닌 다른 서버에 배포하면 IP를 속일 수 있는 가능성이 있습니다. 현재 Vercel에서는 안전하지만, 나중을 위해 주의사항을 기록해두는 게 좋습니다.

- 자체 서버 배포 시 IP 기반 접근 제어 우회 가능성 안내

### S-11. 주문번호 충돌 확률
> **쉬운 설명**: 주문번호가 "TM-날짜-4자리코드" 형식인데, 하루에 주문이 많아지면 같은 번호가 **겹칠 가능성**이 있습니다. 4자리를 6자리로 늘리면 겹칠 확률이 1,300배 줄어듭니다.

- `TM-YYYYMMDD-XXXX` 4자리 영숫자(36^4 = 1,679,616 조합), 일 주문량 증가 시 충돌 위험
- 6자리 이상으로 확장 권장

### S-12. queries.ts categories 빈 배열 접근
> **쉬운 설명**: 상품에 연결된 카테고리가 없는 경우(삭제되었거나 설정 안 됨), 서버가 에러를 일으킬 수 있습니다. "카테고리 없음" 상황을 미리 처리해야 합니다.

- `row.categories`가 빈 배열일 경우 `[0]`이 undefined -> 런타임 에러

### S-13. console.error에 민감한 컨텍스트 로깅
> **쉬운 설명**: 관리자 로그인이 실패하면 서버 로그에 관리자 아이디가 그대로 기록됩니다. 서버 로그에 접근할 수 있는 사람이 관리자 계정명을 볼 수 있습니다.

- 관리자 로그인 실패 시 username 평문 로깅

### S-14. productId UUID 형식 검증 누락
> **쉬운 설명**: 관리자가 핀 번호를 업로드할 때, 상품 ID가 올바른 형식인지 확인하지 않습니다. 잘못된 형식이 들어오면 데이터베이스에서 에러가 나는데, 미리 걸러내는 게 좋습니다.

- `admin/pins/upload`에서 `product_id`를 UUID 검증 없이 쿼리에 사용

## ✅ 잘된 점

팀 전체가 공통으로 인정한 우수 사항:

1. **결제 금액 조작 방지** — 해커가 결제 금액을 10,000원에서 100원으로 바꿔서 보내도, 서버에서 원래 금액과 대조하여 차단합니다
2. **핀 번호 암호화** — 상품권 핀 번호가 데이터베이스에 암호화된 상태로 저장되어, DB가 유출되어도 핀 번호를 알 수 없습니다
3. **입력 검증 일관성** — 모든 서버 기능에서 "올바른 형식인지" 검사를 빠짐없이 수행합니다
4. **XSS 방지** — 사용자 입력을 화면에 그대로 뿌리는 위험한 방식을 전혀 사용하지 않습니다 (악성 스크립트 삽입 차단)
5. **타입 안전성** — 코드에서 데이터 형태를 무시하는 위험한 패턴(`as any`)이 0건입니다
6. **관리자 3중 보안** — 관리자 페이지는 ① 허용된 IP인지 ② 로그인했는지 ③ 세션이 유효한지 3단계로 확인합니다
7. **동시 접근 방지** — 두 사람이 동시에 같은 바우처를 사용하려고 해도, 먼저 처리된 쪽만 성공하고 나머지는 거부됩니다
8. **PG 망취소** — 결제는 됐는데 주문 생성이 실패하면, 자동으로 결제를 취소합니다 (돈만 빠지는 상황 방지)
9. **SQL Injection 방지** — 해커가 입력란에 데이터베이스 명령어를 넣어도, 명령어가 아닌 일반 텍스트로 처리됩니다
10. **파일 경로 조작 방지** — 파일 삭제 시 `../` 같은 경로 조작을 차단하여, 서버의 중요 파일을 삭제할 수 없습니다
